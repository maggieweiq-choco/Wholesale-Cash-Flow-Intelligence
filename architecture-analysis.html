1Product Analysis

Product positioning and value proposition reverse-engineered from the codebase (README, page copy, agent prompts).

"On day 45, you'll be short $23,000 — should you liquidate inventory or take a loan?" — one-line product pitch

Target User

Wholesale business owners / finance managers (small-to-mid wholesale SMBs). Squeezed from both sides: cash locked in warehouse inventory and cash locked in customer payment terms (net 30/60/90). Most currently manage with Excel or gut instinct — both wildly inaccurate.

Core Problem

Owners don't know when a cash gap will appear or how large it will be; they don't know whether to liquidate inventory, chase payments, or borrow first; all three are interdependent, so looking at any one dimension in isolation leads to the wrong decision.

Solution

Claude agents consume sales history, inventory, and AR data to produce a 90-day daily cash flow timeline + dead stock list + collections priority + financing comparison — four outputs linked into one final recommendation.

Business Model

No billing or subscription logic implemented in the current codebase (no pricing/billing files). Monetization strategy has not been built into the product — it remains conceptual.

7Key Architectural Decisions & Rationale

Why DynamoDB + Aurora dual-database instead of just one

DynamoDB absorbs high-frequency, unstructured raw uploads (CSV rows land directly as Record<string,string>, no type enforcement). Aurora holds strongly-typed cleaned tables that agents and the frontend query with structured SQL. This maps cleanly to the hackathon's "database: pick one or combine" bonus — a single database would work, but the dual-database story better fits the technical requirements.

Why RDS Data API instead of direct Postgres connection

lib/aurora.ts uses RDSDataClient + drizzle-orm/aws-data-api/pg; SETUP.md explicitly notes "avoids exhausting the connection pool in serverless." Each Vercel request spins up a fresh serverless function instance; direct Postgres would drain the connection limit quickly. The Data API communicates over HTTP with no connection pool overhead.

Why all agents use tool_choice to force structured output

tools.ts defines tool schemas for each agent; every call passes tool_choice:{type:"tool", name:...} so Claude must respond via the tool, never as free text. This guarantees the return value is always parseable JSON — no defensive parsing needed in the API layer or database write path.

Core bug fixed: agents used to "make up numbers"

lib/queries.ts comment reads verbatim: "This is the fix for the core bug: agents used to prompt Claude with no data, so it invented numbers." All 6 agents now call getCompanyData() before the Claude call and JSON.stringify the real Aurora rows into the prompt — eliminating hallucinated outputs entirely.

Why normalize is "delete all, then re-insert"

lib/etl.ts comment: "Idempotent: wipes the company's existing rows first so re-running an upload doesn't duplicate." Blunt but prevents duplicated sales/inventory figures from re-uploads — a more dangerous failure mode than the latency cost of a full re-insert. A reasonable trade-off within an 11-day sprint.

customers table is derived, not uploaded

Users upload only sales / inventory / invoices / payables CSVs. The customers table — including avg_days_late and payment_score — is computed entirely by lib/etl.ts during normalize from paid_at vs due_at in invoices. This reduces user data-entry burden and keeps the derivation auditable.

Deterministic projection runs without LLM

lib/projection.ts computes the daily cash curve entirely from Aurora data on every page load — no Claude call, no cost, instant response. Real AP due dates from the payables table replace hardcoded demo events when available. The AI forecast is a separate saved run the user triggers manually (or via cron), keeping the always-on timeline free and responsive.

APR normalization for financing options

Liquidating inventory at a 33% haircut is not comparable to a 10% annual loan. The financing agent computes true APR = (cost ÷ raised) × (365 ÷ days) only for time-priced instruments (bank loan, AR finance). Liquidation shows "one-time haircut, not annualized" to avoid the misleading 33% → ~240% APR figure that would make liquidation look far more expensive than it is.

8Known Gaps & Risks (ordered by impact)

GapImpactEvidence
Bedrock embedding is dead code Low (but high explanation cost) — SETUP.md lists Bedrock as an "on-brand" bonus, but the actual product logic has no use for semantic vectors (this is leftover from a rejected direction). If a judge asks about it during the demo, there is no good answer. Global search for embedText finds exactly one occurrence: the definition itself
"Audit Compliance Alert" feature does not exist Low — The README judging criteria table lists this as a differentiator; the codebase has no implementation. Global search for audit / compliance / write-off matches only the README itself
Cron failures are silent Medium — A per-company forecast refresh failure is logged only as -1 in the result object; no retry, no alert, no notification. The next day's dashboard may silently show stale data. app/api/cron/route.ts catch block: results[companyId] = -1
No multi-tenant access control Medium — companyId is an arbitrary string; anyone who knows another company's ID can read their data and financial recommendations. A minimal auth layer (session cookie, token, or fixed demo ID) is needed before a public demo. All API routes accept companyId with no session or auth check
No input validation Low — Zod is installed but unused. A malformed CSV (e.g. non-numeric amount) silently coerces to 0 via Number(d.sold_qty ?? 0) rather than returning an error. lib/etl.ts uses silent numeric coercion throughout
