1产品分析

从代码（README、页面文案、agent prompt）反推出的产品定位与价值主张。

"On day 45, you'll be short $23,000 — should you liquidate inventory or take a loan?" — 产品一句话定位

目标用户

批发商老板 / 财务负责人（中小型 wholesale SMB）。两边被夹：现金压在仓库库存里，现金压在客户账期（net 30/60/90）里。目前多用 Excel 或直觉判断，误差很大。

核心问题

不知道未来现金缺口何时出现、有多大；不知道该先清库存、先催账，还是先借钱；三者互相影响，单看一个维度容易做错决策。

解法

用 Claude agent 把销售历史、库存、应收账款三类数据喂给模型，产出 90 天逐日现金流时间线 + 死货清单 + 催收优先级 + 融资方案对比，四件事联动给一个最终建议。

商业模式

当前代码未实现计费/订阅逻辑（无 pricing/billing 相关文件），定价策略尚未落地到产品中，停留在概念阶段。

7关键架构决策与理由

为什么 DynamoDB + Aurora 双库，而不是只用一个

DynamoDB 吃高频、无结构的原始上传（CSV 行直接落地，字段是 Record<string,string>，不强制类型）。Aurora 存清洗后的强类型表，供 agent 查询和前端展示用结构化 SQL。这正好对应 Hackathon"数据库三选一/组合"的加分点——单数据库也能做完，但双库故事更贴技术要求。

为什么走 RDS Data API 而不是直连 Postgres

lib/aurora.ts 用 RDSDataClient + drizzle-orm/aws-data-api/pg，SETUP.md 明确写"避免 serverless 打爆连接池"——Vercel 的每次请求起一个新的 serverless 函数实例，直连 Postgres 容易耗尽连接数，Data API 走 HTTP，无连接池问题。

为什么 4 个 agent 全用 tool_choice 强制工具调用

tools.ts 定义 4 个 schema，每次调用都 tool_choice:{type:"tool", name:...} 强制 Claude 必须用工具返回，不允许自由文本。这保证返回值永远是可解析的 JSON 结构，前端/数据库写入不需要做容错解析。

核心 bug 修复：agent 曾经"瞎编数字"

lib/queries.ts 注释原话："This is the fix for the core bug: agents used to prompt Claude with no data, so it invented numbers." 现在所有 4 个 agent 调用前都先 getCompanyData() 把 Aurora 里的真实行 JSON.stringify 进 prompt，杜绝模型瞎猜。

为什么 normalize 是"先删全部再插入"

lib/etl.ts 注释写明"Idempotent: wipes the company's existing rows first so re-running an upload doesn't duplicate"。简单粗暴但避免了重复上传导致销售/库存数字翻倍这种更致命的错误，11天冲刺期内是合理取舍。

customers 表是派生表，不是用户直传

用户只上传 sales/inventory/invoices 三种 CSV，customers 的 avg_days_late 和 payment_score 完全由 lib/etl.ts 在 normalize 阶段从 invoices 的 paid_at vs due_at 计算得出——减少了用户填表负担，是个干净的设计。

8已知缺口与风险（按影响排序）

缺口影响证据
主打功能"90天现金流时间线"没有展示页面 高 — README 头一句话卖点（"on day 45 you'll be short $23,000"）评委点开 demo 看不到对应页面，只有 /inventory /receivables /financing 三个子页 grep 全仓库，CashflowChart.tsx 组件存在但零处被 import；app/ 下无 cashflow 或 forecast 路由
无多租户访问控制 中 — companyId 是任意字符串，任何人输入别人的 company id 就能看到别人的数据/财务建议 所有 API route 直接 request.json().companyId，无 session/auth 校验
cron 失败静默 中 — 某公司当天预测刷新失败只记 -1，无重试、无告警，第二天 dashboard 数据可能是过期的 app/api/cron/route.ts 的 catch { results[companyId] = -1 }
loan_scenarios 表实际未被写入 低 — schema 定义了表，但 /api/financing 只把结果 JSON 返回给前端，没有 insert 到这张表，数据不落地 app/api/financing/route.ts 只 return NextResponse.json(recommendation)
Bedrock embedding 是死代码 低（但解释成本高）— SETUP.md 把 Bedrock 列为"on-brand"加分项，但实际产品逻辑完全不需要语义向量（这是从另一个被否决的方向"反馈信号平台"遗留的代码），demo 时如果评委问起容易被问住 全局搜索 embedText 只在定义处出现一次
"审计合规预警"功能名存实亡 低 — README judging criteria 表里写了这条差异化卖点，代码完全没实现 全局搜索 audit/compliance/write-off 只匹配到 README 本身
无输入校验 低 — Zod 已装但未使用，CSV 字段缺失/格式错误时（如金额非数字）会直接产出错误数据而不报错 lib/etl.ts 用 Number(d.sold_qty ?? 0) 静默兜底成 0，不抛错
